  <action>
**payment.service.ts** -- modify `initiatePayment`:

1. Add `couponCode?: string` parameter (4th param, optional).
2. After finding the plan, calculate discounts:
   - If couponCode provided, call `validateCoupon(couponCode, userId)` to get couponId and discountPercentage.
   - Calculate `couponDiscount = Math.round(plan.price * (discountPercentage / 100))`.
   - `afterCoupon = plan.price - couponDiscount`.
   - Fetch user's `referralCreditBalance`. Calculate `creditsUsed = Math.min(referralCreditBalance, afterCoupon)`.
   - `finalAmount = afterCoupon - creditsUsed`.

3. If `finalAmount <= 0` (KES 0 path):
   - Use `prisma.$transaction` to atomically:
     a. Create Payment record with amount: 0, status: "SUCCESS", method: "CREDITS", couponId, couponDiscount, referralCreditsUsed: creditsUsed, idempotencyKey: crypto.randomUUID().
     b. If creditsUsed > 0, decrement user.referralCreditBalance by creditsUsed.
     c. If couponId, call redeemCoupon(tx, couponId, userId, payment.id).
     d. Call activateSubscription(tx, payment.id).
   - Return `{ paymentId, checkoutRequestId: "CREDITS", isExisting: false }`.

4. If `finalAmount > 0` (normal M-Pesa path):
   - Create PENDING payment with `amount: finalAmount`, `discount: couponDiscount + creditsUsed`, `couponId`, `couponDiscount`, `referralCreditsUsed: creditsUsed`.
   - Initiate STK Push with `finalAmount` (not plan.price).
   - On STK Push success, if creditsUsed > 0, update user's referralCreditBalance (decrement creditsUsed). Do this OUTSIDE the STK push (credits are consumed at initiation, refunded on failure).
   - Actually, better approach: defer credit deduction to callback success. Store creditsUsed on the Payment record. In processCallback success path, THEN deduct credits and redeem coupon.

REVISED APPROACH for safety (no credit loss on failed payments):
- At initiation: validate coupon, calculate amounts, store couponId/couponDiscount/referralCreditsUsed on PENDING payment. Do NOT deduct credits yet.
- At callback success: inside the $transaction, deduct referralCreditsUsed from user.referralCreditBalance, call redeemCoupon for coupon, then activateSubscription.
- For KES 0 path: do everything atomically in one transaction at initiation time (no callback needed).

**Modify `processCallback`**:
Inside the success transaction (ResultCode === 0), after updating payment to SUCCESS:
1. If `payment.referralCreditsUsed > 0`, decrement user's referralCreditBalance by that amount.
2. If `payment.couponId`, call redeemCoupon(tx, payment.couponId, payment.userId, payment.id).
3. Call `grantReferralCredit(tx, payment.userId)` to check if this user was referred and grant credit to referrer on first successful payment.
4. Then call activateSubscription(tx, payment.id) as before.

Note: For grantReferralCredit, also check that payment is the referee's FIRST successful payment (count payments where userId = refereeId AND status = SUCCESS, if count === 1 including current, then grant).

**payment.validators.ts** -- update `initiatePaymentSchema`:
- Add `couponCode: z.string().min(1).max(50).optional()` to the schema.

**auth.service.ts** -- modify `register` function:
- Change the referral code lookup to check BOTH tables:
  1. First check `prisma.user.findUnique({ where: { referralCode: data.referralCode } })`.
  2. If not found, check `prisma.adminInviteCode.findFirst({ where: { code: data.referralCode, isActive: true } })`.
  3. If admin invite code found, verify `usedCount < maxUses`. If exceeded, throw AuthError("INVITE_CODE_EXHAUSTED", "This invite code has been fully used", 400).
  4. If neither found, throw existing INVALID_REFERRAL_CODE error.
  5. In the transaction: if it was an admin invite code, increment `adminInviteCode.usedCount` and do NOT create a Referral record. If it was a user referral code, create the Referral record as before.

Use ESM imports with .js extensions throughout.
