  <action>
    Update the Content model in schema.prisma:
    - Add `cast String[]` field (after categories)
    - Add `director String?` field (after cast)

    Add a new Category model:
    ```
    model Category {
      id        String   @id @default(uuid())
      name      String   @unique
      slug      String   @unique
      createdAt DateTime @default(now())

      @@index([name])
    }
    ```

    The Category model is the source of valid category names. Content.categories remains String[] for simplicity (no join table). Admin creates categories in the Category table; content forms populate dropdowns from it; selected values stored as strings in categories[].

    Run the migration using manual SQL approach (prisma migrate dev fails in non-interactive terminal):
    1. Run `npx prisma migrate diff --from-migrations-directory prisma/migrations --to-schema-datamodel prisma/schema.prisma --script` to generate the SQL
    2. Create migration directory manually: `mkdir -p prisma/migrations/{timestamp}_add_cast_director_category`
    3. Write the SQL to `migration.sql`
    4. Run `npx prisma migrate deploy`
    5. Run `npx prisma generate` to update the client

    IMPORTANT: All commands run from the api/ directory. PrismaClient is imported from "../generated/prisma/client.js".
  </action>
  <verify>
    Run `cd api && npx prisma migrate status` -- should show all migrations applied.
    Run `cd api && npx prisma validate` -- should pass.
    Run `cd api && npx tsc --noEmit` -- should compile with no errors.
  </verify>
  <done>Content model has cast String[] and director String? fields. Category model exists with id, name (unique), slug (unique), createdAt. Migration applied successfully.</done>

  <action>
    1. Add requireAdmin middleware to api/src/middleware/auth.middleware.ts:
       - Export a new `requireAdmin` function with same signature as requireAuth
       - Check `req.user?.role !== 'ADMIN'` and return 403 with `{ error: { message: 'Admin access required', code: 'FORBIDDEN' } }`
       - Call `next()` if admin
       - This middleware is ALWAYS used after requireAuth in the chain: `requireAuth, requireAdmin`

    2. Create api/src/validators/admin.validators.ts:
       - `adminLoginSchema`: z.object({ email: z.string().email(), password: z.string().min(1) })

    3. Create api/src/services/admin.service.ts:
       - `adminLogin(email: string, password: string)` function
       - Find user by email, verify password with argon2, check user.role === 'ADMIN'
       - If not admin, throw an error with message "Invalid credentials" (don't reveal that user exists but isn't admin)
       - Create a session (reuse createSession from session.service.ts) with device info from request
       - Return { user, token } (same pattern as auth.service login but simpler -- no device limit check needed for admin)
       - Import argon2.verify, prisma, createSession, signToken from existing services

    4. Create api/src/routes/admin.routes.ts:
       - POST /login -- validates body with adminLoginSchema, calls adminLogin, returns 200 with { user: { id, email, name, role }, token }
       - Use inline try/catch pattern (same as auth.routes.ts)
       - Export adminRouter

    5. Update api/src/routes/index.ts:
       - Import adminRouter
       - Register: `app.use("/api/admin", adminRouter)`

    6. Create api/prisma/seed.ts:
       - Import PrismaClient from "../src/generated/prisma/client.js" and argon2

       - Use upsert to be idempotent
       - Log the credentials to console after creation

    7. Update api/package.json:
       - Add `"prisma": { "seed": "npx tsx prisma/seed.ts" }` to package.json
       - Add tsx as dev dependency if not present: `npm install -D tsx`

    ESM RULES: All imports must use .js extensions. PrismaClient from "../generated/prisma/client.js".
