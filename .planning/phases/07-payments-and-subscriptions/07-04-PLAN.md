2. Create api/src/jobs/reconciliation.job.ts following the pattern from sessionCleanup.job.ts:
   - Import cron from "node-cron"
   - Import prisma from "../config/database.js"
   - Import getMpesaClient from "../config/mpesa.js"
   - Import processCallback from "../services/payment.service.js" (or a dedicated reconciliation handler)

   Export `startReconciliationJob(): void`:
   - Schedule with `cron.schedule("*/2 * * * *", async () => { ... })`
   - Find PENDING payments WHERE:
     - createdAt < 2 minutes ago (give callback time to arrive)
     - reconciliationAttempts < 5 (max 5 retries)
     - checkoutRequestId is not null (STK Push was initiated)
   - For each pending payment:
     - Increment reconciliationAttempts first (prevents concurrent processing)
     - Call getMpesaClient().querySTKStatus(payment.checkoutRequestId)
     - ResultCode 0 (success):
       - Build a synthetic callback-like payload and call the payment processing logic
       - OR directly: update payment to SUCCESS + create subscription (using same atomic $transaction pattern)
       - Log: `[RECONCILIATION] Resolved payment ${payment.id} as SUCCESS`
     - Terminal error codes (1, 1032, 1037):
       - Update payment to FAILED with resultCode and resultDesc
       - Log: `[RECONCILIATION] Resolved payment ${payment.id} as FAILED: ${result.ResultDesc}`
     - Non-terminal (1001 system busy):
       - Leave as PENDING for next cron run
     - If reconciliationAttempts reaches 5 and still not resolved:
       - Mark as FAILED with resultDesc "Reconciliation timeout"
       - Log: `[RECONCILIATION] Payment ${payment.id} failed after 5 attempts`
   - Wrap each payment in try/catch (one failure shouldn't stop others)
   - Outer try/catch for the entire job
   - Only log when there's something to do (quiet runs when no pending payments)
   - Log startup: `[RECONCILIATION] Scheduled every 2 minutes for lost M-Pesa callbacks`

   Helper: `isTerminalResultCode(code: number): boolean` -- returns true for 0, 1, 1032, 1037, 9999
